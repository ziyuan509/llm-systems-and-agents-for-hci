# Related Work Map

## Cluster 1: Pervasive AI, Bystander Privacy, and Shared-Space Sensing

Use this cluster to position the project within UbiComp-style problems where AI agents continuously sense, remember, and act in shared physical environments.

Key question:

- How should ambient AI systems expose privacy boundaries around bystanders, memory, disclosure, and situated action?

Why it matters:

- Pervasive AI agents may operate in shared spaces such as studios, homes, classrooms, offices, and wearable contexts.
- These agents may capture information from people who are not the primary user.
- Privacy control must therefore account for bystanders, shared context, and ambiguous consent.
- The project uses delegation contracts to make these boundaries visible and revisable by ordinary users.

Potential concepts to connect:

- bystander privacy
- privacy-preserving sensing
- shared-space computing
- ambient intelligence
- context-aware systems
- situated consent
- disclosure boundaries

## Cluster 2: End-User Control of Generative and Agentic Systems

Use this cluster to connect with prompt editing, workflow customization, mixed-initiative control, and end-user programming.

Key question:

- Why are policy objects a better control surface than raw prompt edits for ordinary users?

Why it matters:

- Ordinary users should not need to rewrite prompts every time an agent crosses a boundary.
- Stable, revisable policy objects can support longer-term governance.
- Delegation contracts turn abstract user preferences into interaction objects that can be inspected and repaired.

Potential concepts to connect:

- end-user programming
- mixed-initiative interaction
- prompt-based control
- user intent specification
- human-AI collaboration
- control surfaces for AI systems

## Cluster 3: Agent Transparency, Debugging, and Runtime Steering

Use this cluster to position against systems that emphasize traces, states, and developer-facing intervention.

Key question:

- Why is delegation governance a different interaction problem from trace visibility?

Why it matters:

- Trace views can expose what an agent did, but they may not help ordinary users define what the agent should be allowed to do.
- Developer-facing debugging tools are often too complex for everyday privacy decisions.
- Delegation contracts focus on user-facing boundaries rather than raw execution details.

Potential concepts to connect:

- AI transparency
- agent traces
- runtime steering
- debugging interfaces
- explainable AI
- user-facing accountability

## Cluster 4: Privacy, Safety, and Policy Enforcement in Agents

Use this cluster to motivate why governable surfaces include memory, tools, disclosure, and escalation.

Key question:

- Which backend controls should become user-visible interaction objects?

Why it matters:

- Agent safety is not only a backend enforcement problem.
- Users need to understand and revise how privacy and safety rules shape agent behavior.
- The interface should expose key boundaries around sensing, memory, tools, disclosure, and escalation.

Potential concepts to connect:

- privacy policies
- memory control
- tool-use permissions
- disclosure control
- escalation policies
- user consent
- accountability mechanisms

## Cluster 5: Calibrated Reliance and Ask-back Design

Use this cluster to frame when agents should act autonomously and when they should ask the user.

Key question:

- How can ask-back moments help users maintain calibrated reliance on AI agents?

Why it matters:

- If an agent asks too often, users may ignore it.
- If an agent asks too rarely, it may violate privacy or safety expectations.
- Ask-back moments can become governance moments where users clarify future boundaries.

Potential concepts to connect:

- calibrated trust
- reliance
- overtrust
- undertrust
- interruption design
- confirmation dialogs
- uncertainty communication
- human-in-the-loop AI

## Cluster 6: Longitudinal Personal Agent Governance

Use this cluster to frame ongoing revision, accountability, and sustained agent use over time.

Key question:

- How should delegation contracts evolve across tasks instead of only within a session?

Why it matters:

- Privacy boundaries are not fixed.
- Users may only understand their preferences after seeing an agent make a mistake.
- Repair should update future behavior, not only correct a single output.
- Delegation contracts can support long-term governance by turning breakdowns into revised rules.

Potential concepts to connect:

- longitudinal interaction
- personal AI agents
- memory over time
- preference learning
- repair and recovery
- policy evolution
- accountable adaptation
